# V3 Viewer UX Audit — Concept vs Live

**Captured:** 2026-05-01 · branch `feature/taskmaster-v3` · tip `184e9a1`
**Source concepts:** `.superpowers/brainstorm/16245-1777231623/content/` (32 mockups; latest variant per screen used)
**Method:** per-screen comparison of concept HTML mockup vs live viewer screen JS/CSS, focusing on interactivity (drill-in, search, hover), filter affordances, and surface tokens.

## Executive verdict

The L1–L3 control-consistency work and the just-landed Pass A made screens **structurally consistent** but did not fix the **interactive gaps**: most cards don't drill in, search is missing from Lessons and Issues entirely, and Sessions has a search input that fires nothing. The current "polish" passes were sanding edges off cards that aren't even clickable.

The *visual* surface stepping is mostly correct across screens — only Lessons uses the wrong token (`--bg-card` where concept has `--bg-panel`). The bigger issue is **interaction debt**, not visual debt.

## Patterns across screens

### Drill-in is missing on 3 screens
- **Lessons** — `lessons.js` registers no click handlers on lesson cards
- **Issues** — only nested task pills route; clicking the card body does nothing
- **Dashboard** — `createBoardSurface` mounts task cards but doesn't wire clicks (the kanban `card.js` handler isn't reused)

### Search is missing on 2 screens, dead on 1
- **Lessons** — concept has `<input placeholder="Search lessons & anchors…">`; live has nothing
- **Issues** — concept has `<input placeholder="Search issues, components…">`; live has nothing
- **Sessions** — `tmSearch` mounted, fires `viewer:sessions-search` event, but `renderTimeline` ignores it. Worst-of-both: looks interactive, isn't.

### Filter chip-rows are missing on 2 screens
- **Lessons** — scope chips (All / CSS / Git / Plugins / Schema / Workflow) absent
- **Issues** — component chips (All / viewer / cli / plugins) absent

### Surface tokens are mostly right
The concept's stepping (`--bg-shell` < `--bg-panel` < `--bg-card` < `--bg-deep`) is honored on Dashboard, Kanban, Issues, Sessions, Recap, Task Detail, Auto Mode. The one outlier is **Lessons**: cards use `var(--bg-card)` where concept uses `--bg-panel` — the cards are visually too bright by one surface step.

### What's NOT actually broken
- Hover-no-transform discipline is upheld across the board (LSN-19 wins)
- Auto-status pill survives every navigation pair
- Topbar primitive system from L1–L3 is solid and consistent
- Card click → task detail works on Kanban and Sessions; the issue is that other screens don't reuse the pattern

## Re-sequenced polish passes

**Old plan (in handoff):** Pass A (Lessons polish, done) → Pass B (overflow audit) → Pass C (`.tm-card` unification).

**Issue with old plan:** B and C are visual/structural — they don't address the interaction gaps. Polishing card chrome before cards click feels backwards.

**Proposed new sequence:**

### Pass B′ — Interaction backbone (drill-in + search)
Goal: every card on every screen does what the concept implies it does.

- **B1 — Drill-in:** wire click-to-detail on lesson cards, issue cards, dashboard task cards. Lessons + Issues need a detail view (right-rail panel or full screen — TBD). Dashboard task cards reuse the existing `#/task/:id` route.
- **B2 — Search:** add search to Lessons + Issues using the existing `tmSearch` primitive. Make Sessions search actually filter the timeline (or remove it).
- **B3 — Filter chips:** scope chips on Lessons (concept lists 6 categories), component chips on Issues (concept lists 4). May require server-side categorization or be handled client-side from existing fields.
- **B4 — Lessons surface fix:** swap `--bg-card` for `--bg-panel` on `.lesson-card`. One line.

### Pass C′ — Visual consistency + card primitive
Goal: a unified `.tm-card` that codifies the conventions B′ exposes.

- **C1 — `.tm-card` primitive:** use Kanban's `.card-task` as canon. Variants for issue / lesson / recap / sessions-handover. Define overflow (title clamp, chip-row wrap, file-path truncation).
- **C2 — Card overflow audit:** narrow-width regressions (sidebar collapsed, smaller viewport).
- **C3 — Right-rail consistency:** Sessions has a working detail rail; Lessons and Issues should reuse the same rail component once B1 lands.

### Pass D — Visual polish nits
The concept fidelity items that don't block interaction.

- **D1 — Dashboard:** auto-strip session timer, briefing strip phase pips
- **D2 — Recap:** rebuild the session picker with current-recap identity (id + title + date), keep ‹/› nav arrows
- **D3 — Sessions:** Lanes / By-Task views (currently stubs) — concept-driven implementations
- **D4 — Issues:** SVG hex severity glyphs (concept uses inline SVG defs)
- **D5 — Sessions timeline dimming:** when right-rail is open, non-selected items dim to 0.5 opacity per concept

### Pass E — Pre-existing follow-ups (already flagged in earlier handoffs)
- XSS fix in lessons.js By-Anchor view (review-gate Important #1)
- View-toggle test class assertion tightening
- Add empty-summary negative-case test for lesson-card
- `--epic-viewer` token: define in tokens.css or replace usages with `--accent-blue`
- Lesson reinforce stale-comment cleanup

## Open questions (need user input before B1)

1. **Detail view shape for Lessons and Issues:** right-rail (matches Sessions pattern) or full-screen (matches Task Detail)? Concept doesn't show explicit detail mockups for either — drill-in is implied by `cursor:pointer`.
2. **Scope filter source for Lessons (B3):** concept categories are CSS/styling, Git/branches, Plugins, Schema, Workflow. These don't exist in the lesson schema. Options: (a) add a `category` field to lesson frontmatter and migrate fixtures, (b) infer from `triggers.files` patterns, (c) drop the scope row and accept the deviation.
3. **Sessions search (B2):** make it actually filter, or remove the dead input? "Make it work" is the right answer if implementation is cheap; otherwise remove and re-add later.
4. **Dashboard task-card click (B1):** route to `#/task/:id` (kanban behavior) or open a quick-view popover? Concept implies the former.

## Resume here

After user signs off on the proposed sequence + answers the 4 open questions, start B1 (drill-in). That's the highest-leverage single change — it makes 3 screens functionally complete, after which B2/B3 are quick wins on top of working navigation.
